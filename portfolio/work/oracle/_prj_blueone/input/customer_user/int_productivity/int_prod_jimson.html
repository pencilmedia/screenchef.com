HAND DATA ENTRY CASES

1. Describe the process you follow to enter PARTS by hand. What works well in Agile? What about Agile makes this difficult?

For the IC Design case (see below), engineers are free to create parts and BOMs if parts are not released and then add to an ECO for release.
What works well is it that it's easy with lots of flexibility. Don't need a formal parts request process. The downside is part proliferation [SG - but they have not really seen this in practice].

2. Describe the process you follow to enter / modify BOMs by hand. What works well in Agile? What about Agile makes this difficult?

For IC (process oriented, deep multi-level BOMs) we create these by hand in the Agile UI. [SG - 8 levels deep with lots of Documentation]

For PCB BOMs, we import these to Agile via Agile Import due to the fact there are 100s of parts.

3. Describe your change management process? How are BOMs, AMLs, Drawings changed during the ECO process? What works well in Agile? What about Agile makes this difficult?

Life cycle changes are made via ECO if BOM / Rev is going to change / MCO if no revision / BOM change. Agile is missing the ability to control P2, P3 attributes by revision. Find the save-as feature to be very useful.

AGILE 921 VS. AGILE 9

1. What key productivity features did you GAIN coming to 921 from 9?

How quickly we're able to do things. A9 was slow for things like creating records (items) and BOMs. 921 is much faster. File upload is much better. 

2. What key productivity features did you LOSE coming to 921 from 9?

Reporting. Lost the ability to sort/group on list fields. Use list fields to store critical information and can no longer report on it. Agile's reporting is very weak. 

3. Have you changed how you use Agile in 921 compared to 9? How?

Users are more enthusiastic about using Agile. I'm receiving less complaints. In A9 we couldn't save 600MB / 1GB files. Now we can.

[SG notes] - 921 is faster to create items or BOMs.  Files upload faster.  Better speed = productivity gains.

[PGC] PGC is far better but still inadequate.
